Jenkins CI/CD to AWS EKS Runbook

A polished, practical runbook for starting with an empty Jenkins server and finishing with a working deployment pipeline to Amazon EKS.

Local code→GitHub→Jenkins→Docker Hub→AWS EKS→kubectl verify

1. How everything connects

This project is not magic. It works because each tool has credentials, configuration, and a clear responsibility.

ToolJobConnection
Local machineYou edit code and push it.Uses Git remote pointing to GitHub.
GitHubStores code and branches.Sends webhook events to Jenkins.
JenkinsRuns CI/CD pipeline from Jenkinsfile.Uses GitHub, Docker Hub, and AWS credentials.
Docker HubStores application Docker images.Jenkins pushes image tags here.
AWS IAMControls AWS permissions.Dedicated IAM user lets Jenkins access EKS.
EKSRuns Kubernetes cluster.Jenkins and local kubectl both talk to the EKS API server.
kubectlRemote control for Kubernetes.Uses kubeconfig to know which cluster to contact.
Senior mental model

Jenkins deploys to the remote EKS cluster. Your local terminal does not read Jenkins; it reads the same EKS cluster directly through kubeconfig.

3. Create and use the deployment branch locally

Use a separate branch for this Kubernetes deployment work. This keeps the deployment experiment clean and easy to manage.

bash
git fetch origin
git checkout -b deploy-to-k8s origin/deploy-to-k8s
git branch

Why: Jenkins will build this branch, so your Jenkinsfile and Kubernetes manifests must be pushed there.

4. Create a dedicated AWS IAM user for Jenkins

Do not use your personal AWS admin user for Jenkins. Create a dedicated IAM user with limited permissions.

1

Create IAM user

AWS Console path:

IAM → Users → Create user
User name: jenkins-eks-deployer
Console access: No
2

Add a limited IAM policy

This policy allows Jenkins to describe only your EKS cluster. That is needed for aws eks update-kubeconfig.

json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowDescribeSpecificEksCluster",
      "Effect": "Allow",
      "Action": [
        "eks:DescribeCluster"
      ],
      "Resource": "arn:aws:eks:eu-central-1:YOUR_ACCOUNT_ID:cluster/jennifer-demo-cluster"
    }
  ]
}
3

Create access keys

IAM → Users → jenkins-eks-deployer → Security credentials → Create access key
Save: Access Key ID + Secret Access Key
Least privilege principle

Start with only the permissions Jenkins needs. For this project, the IAM side needs eks:DescribeCluster. Kubernetes permissions are handled by EKS Access Entry later.

5. Create the EKS cluster with eksctl

Create the cluster before adding the EKS Access Entry, because the cluster must exist first.

5.1 Create cluster.yaml

bash
nano cluster.yaml
yaml
apiVersion: eksctl.io/v1alpha5
kind: ClusterConfig

metadata:
  name: jennifer-demo-cluster
  region: eu-central-1
  version: "1.30"

managedNodeGroups:
  - name: jennifer-demo-nodes
    instanceType: t3.medium
    desiredCapacity: 2
    minSize: 1
    maxSize: 3
    volumeSize: 20
    labels:
      role: worker
    tags:
      nodegroup-role: worker

5.2 Create the cluster

bash
eksctl create cluster -f cluster.yaml

Why: eksctl creates the EKS control plane, managed node group, EC2 worker nodes, and supporting AWS resources.

5.3 Connect your local kubectl

bash
aws eks update-kubeconfig \
  --region eu-central-1 \
  --name jennifer-demo-cluster

kubectl get nodes
kubectl get pods -A

Why: This lets your laptop talk to the EKS Kubernetes API server.

6. Give Jenkins IAM user access inside EKS

IAM permission lets Jenkins call AWS APIs. EKS Access Entry lets Jenkins use Kubernetes API through kubectl.

EKS → Clusters → jennifer-demo-cluster → Access → Create access entry
Principal ARN: arn:aws:iam::YOUR_ACCOUNT_ID:user/jenkins-eks-deployer
Type: Standard

Associate an access policy

Policy: AmazonEKSClusterAdminPolicy
Access scope: Cluster
Learning/demo scope

Cluster admin is acceptable for a learning lab. In production, reduce this to namespace-level access and only the Kubernetes actions Jenkins needs.

13. Create Kubernetes files in the repo

13.1 Create the folder

bash
mkdir -p k8s

13.2 deployment.yaml

bash
nano k8s/deployment.yaml
yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: java-maven-app
spec:
  replicas: 2
  selector:
    matchLabels:
      app: java-maven-app
  template:
    metadata:
      labels:
        app: java-maven-app
    spec:
      containers:
      - name: java-maven-app
        image: mujuules01/demo-app:latest
        ports:
        - containerPort: 8080

Why: The Deployment runs your app, keeps two pods alive, and lets Kubernetes restart failed pods.

13.3 service.yaml

bash
nano k8s/service.yaml
yaml
apiVersion: v1
kind: Service
metadata:
  name: java-maven-app-service
spec:
  type: LoadBalancer
  selector:
    app: java-maven-app
  ports:
  - protocol: TCP
    port: 80
    targetPort: 8080

Why: The Service exposes the pods. type: LoadBalancer asks AWS to create an external load balancer.

14. Final Jenkinsfile

Replace YOUR_REPO_NAME with your real GitHub repository name.

groovy
#!/usr/bin/env groovy

pipeline {
    agent any

    tools {
        maven 'maven-3.9'
    }

    environment {
        DOCKER_REPO = 'mujuules01/demo-app'
        GITHUB_REPO = 'YOUR_REPO_NAME'
        GITHUB_REPO_OWNER = 'mjulestek'
        GIT_BRANCH_TO_PUSH = 'deploy-to-k8s'
        AWS_REGION = 'eu-central-1'
        EKS_CLUSTER_NAME = 'jennifer-demo-cluster'
    }

    stages {
        stage('increment version') {
            steps {
                script {
                    echo 'incrementing the version...'

                    sh 'mvn build-helper:parse-version versions:set -DnewVersion=\\${parsedVersion.majorVersion}.\\${parsedVersion.minorVersion}.\\${parsedVersion.nextIncrementalVersion} versions:commit'

                    env.IMAGE_TAG = sh(
                        script: 'mvn help:evaluate -Dexpression=project.version -q -DforceStdout',
                        returnStdout: true
                    ).trim()

                    env.DOCKER_IMAGE = "${DOCKER_REPO}:${IMAGE_TAG}"

                    echo "New app version is: ${IMAGE_TAG}"
                    echo "Docker image will be: ${DOCKER_IMAGE}"
                }
            }
        }

        stage('build app') {
            steps {
                script {
                    echo 'building the application...'
                    sh 'mvn package'
                }
            }
        }

        stage('build image') {
            steps {
                script {
                    echo 'building the docker image...'

                    withCredentials([usernamePassword(credentialsId: 'docker-hub-repo', passwordVariable: 'DOCKER_PASS', usernameVariable: 'DOCKER_USER')]) {
                        sh 'docker build -t $DOCKER_IMAGE .'
                        sh 'echo $DOCKER_PASS | docker login -u $DOCKER_USER --password-stdin'
                        sh 'docker push $DOCKER_IMAGE'
                    }
                }
            }
        }

        stage('deploy to EKS') {
            steps {
                script {
                    echo 'deploying application to EKS...'

                    withCredentials([
                        string(credentialsId: 'aws-access-key-id', variable: 'AWS_ACCESS_KEY_ID'),
                        string(credentialsId: 'aws-secret-access-key', variable: 'AWS_SECRET_ACCESS_KEY')
                    ]) {
                        sh 'export AWS_PAGER="" && aws sts get-caller-identity'
                        sh 'export AWS_PAGER="" && aws eks update-kubeconfig --region $AWS_REGION --name $EKS_CLUSTER_NAME'
                        sh 'kubectl get nodes'
                        sh 'kubectl apply -f k8s/'
                        sh 'kubectl set image deployment/java-maven-app java-maven-app=$DOCKER_IMAGE'
                        sh 'kubectl rollout status deployment/java-maven-app'
                    }
                }
            }
        }

        stage('commit version update to git repo') {
            steps {
                script {
                    echo 'commit version update to git repo...'

                    withCredentials([usernamePassword(credentialsId: 'github-credentials', passwordVariable: 'GITHUB_TOKEN', usernameVariable: 'GITHUB_USER')]) {
                        sh 'git config --global user.email "jenkins@example.com"'
                        sh 'git config --global user.name "jenkins"'

                        sh 'git status'
                        sh 'git branch'

                        sh 'git remote set-url origin https://$GITHUB_USER:$GITHUB_TOKEN@github.com/$GITHUB_REPO_OWNER/$GITHUB_REPO.git'
                        sh 'git remote -v'

                        sh 'git fetch origin $GIT_BRANCH_TO_PUSH'
                        sh 'git checkout -B $GIT_BRANCH_TO_PUSH origin/$GIT_BRANCH_TO_PUSH'

                        sh 'git add .'
                        sh 'git commit -m "jenkins increment version [skip ci]" || true'
                        sh 'git pull --rebase origin $GIT_BRANCH_TO_PUSH'
                        sh 'git push origin HEAD:$GIT_BRANCH_TO_PUSH'
                    }
                }
            }
        }
    }
}

What the pipeline builds

Maven package → Docker image → Docker Hub image tag.

What the pipeline deploys

Kubernetes manifests → updated Deployment image → rollout status.

16. Troubleshooting

AWS pager error

Error: No such file or directory: less

bash
export AWS_PAGER=""
aws configure set cli_pager ""

Jenkins cannot find AWS credentials

Check IDs exactly match:

aws-access-key-id
aws-secret-access-key

kubectl fails in Jenkins

bash
aws sts get-caller-identity
aws eks update-kubeconfig --region eu-central-1 --name jennifer-demo-cluster
kubectl get nodes

Also confirm EKS Access Entry for jenkins-eks-deployer.

Docker login fails

Check Docker credential ID:

docker-hub-repo

Git push rejected

bash
git fetch origin $GIT_BRANCH_TO_PUSH
git checkout -B $GIT_BRANCH_TO_PUSH origin/$GIT_BRANCH_TO_PUSH
git pull --rebase origin $GIT_BRANCH_TO_PUSH
git push origin HEAD:$GIT_BRANCH_TO_PUSH

Pod ImagePullBackOff

bash
kubectl describe pod POD_NAME

Check image name, image tag, Docker Hub repo visibility, and whether the image was pushed.

aws elbv2 describe-load-balancers --region eu-central-1
